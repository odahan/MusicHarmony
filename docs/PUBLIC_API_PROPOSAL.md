# Enaxos Music Theory — .NET 8 Public API Contract

Status: current contract aligned with the implementation
Assembly: `Enaxos.MusicTheory`
Framework: `net8.0`
Documented version: `1.2.5`

This API favors immutable values, pure calculations, explicit errors, and domain names. The signatures below form the maintained public contract for the library. The generated API documentation remains the most detailed source for XML comments and exact signatures.

## 1. Organization

```text
Enaxos.MusicTheory
├── Primitives       letters, accidentals, pitches, notes
├── Intervals        intervals and transposition
├── Formulas         shared relative degrees
├── Scales           formulas, scales, structures, relations
├── Tonality         keys, signatures, harmonic functions, scale chords
├── Harmony          chords, voicings, and inversions
├── Analysis         chord and scale recognition
├── Tuning           frequencies and temperaments
├── Midi             explicit MIDI conversions
├── Circle           circle of fifths and relationships
├── Presentation     formatting with no global localization dependency
└── Geometry         normalized circle geometry
```

The package has no dependency on a UI framework. WPF, WinUI, MAUI, Avalonia, Blazor, or other applications adapt coordinates and labels to their own types.

## 2. API Principles

- `nullable` is enabled.
- Immutable value types are used for primitives.
- Immutable `sealed` types are used for aggregates that contain collections.
- Output collections are `IReadOnlyList<T>`; no mutable collection is exposed.
- Incoming `IEnumerable<T>` values are defensively copied before validation.
- Aggregates with collections use explicit equality; they do not rely on the reference equality of a list generated by a default `record`.
- There is no string constructor. `Parse` and `TryParse` are explicit.
- There is no ambiguous Boolean such as `preferSharp`; enums or strategies are used.
- No global state influences calculations. Only the default display terminology is global, atomic, and locally overridable.
- Operations that need context receive it as a parameter.
- Formatting, tuning, and MIDI services remain separate from the harmonic model.

### Scale definitions and recognition catalogs

The library exposes **52 distinct public scale definitions**: 21 principal modes (7 major, 7 harmonic-minor, and 7 ascending-melodic-minor), 2 standard pentatonics, 25 exotic definitions, and 4 directly constructible parent collections: `Major`, `NaturalMinor`, `HarmonicMinor`, and `MelodicMinorAscending`.

`ModeCatalog.Standard.AllWithPentatonicAndExoticScales` is the maximal recognition catalog. It contains **48 distinct candidates**: 21 principal modes, 2 pentatonics, and 25 exotic definitions. The four direct parent collections deliberately do not appear in it because their corresponding first modes already represent the same collections for recognition. They remain available from `StandardScales` for direct construction and domain-specific APIs.

`ExoticScales.All` centralizes the 25 exotic definitions. `DiminishedWholeHalf` and `DiminishedHalfWhole` are compatibility aliases for `OctatonicWholeHalf` and `OctatonicHalfWhole`; they do not add definitions or recognition candidates.

Usage:

```csharp
var scale = Scale.Create(
    SpelledPitch.Parse("F#"),
    StandardScales.Major);

// F# G# A# B C# D# E#
```

Identifiers such as `scale.major` are stable and not localized. A formatter turns the identifier into a French, English, or other label.

Examples of labels for the same definition:

```text
mode.major.5 -> "mode de sol" in French
mode.major.5 -> "Mixolydian" in American terminology
```

Rotations of harmonic minor and melodic minor scales also have stable identifiers. The presentation catalog may provide several aliases when terminology is not universal.

```csharp

{
    public Scale Scale { get; }

    // Normalized over the returned set; not a statistical probability
    // calibrated from a corpus.
    public double RelativeProbability { get; }

    public IReadOnlyList<SpelledPitch> MatchedPitches { get; }
    public IReadOnlyList<SpelledPitch> MissingPitches { get; }
    public IReadOnlyList<SpelledPitch> OutsidePitches { get; }
    public IReadOnlyDictionary<string, double> ScoreFactors { get; }

public static class ScaleRecognizer
{
    public static IReadOnlyList<ScaleRecognitionCandidate> FindCandidates(
        IEnumerable<Note> notes,
        ScaleRecognitionOptions? options = null);

    public static IReadOnlyList<ScaleRecognitionCandidate> FindCandidates(
        Chord chord,
        ScaleRecognitionOptions? options = null);
}
```

Results are sorted by descending `Score`, then by key signature, tonic, and mode identifier so ordering remains deterministic.

`RelativeProbability` is calculated by softmax normalization of returned scores. `ProbabilityTemperature` controls the spread between candidates and must be strictly positive.

By default, the recognizer uses the main standard catalog. `IncludePentatonicCandidates` and `IncludeExoticCandidates` extend that default catalog. An explicit `Catalog` replaces that choice and becomes the exhaustive list of candidate definitions.


`CircleDistance` preserves both paths. Opposite positions have six steps in each direction; the API therefore does not arbitrarily choose a sign.

Usage for musical relationships:

```csharp
var circle = CircleOfFifths.Standard;
var eb = circle.Find(MusicalKey.Major(SpelledPitch.Parse("Eb")));
var neighbors = circle.GetNeighbors(eb);

MusicalKey dominant = neighbors.Dominant.Primary.Major;       // Bb major
MusicalKey subdominant = neighbors.Subdominant.Primary.Major; // Ab major
MusicalKey relative = eb.Primary.RelativeMinor;               // C minor
```

The three conventional double spellings remain accessible:

```text
index 5 : B / Cb
index 6 : F# / Gb
index 7 : C# / Db
```

The strategy removes no spelling; it only chooses `Primary`.

## 11. Errors

Programming errors:

- `ArgumentOutOfRangeException` for an invalid index, radius, or frequency;
- `ArgumentException` for an inconsistent definition;
- `InvalidOperationException` only when a valid state cannot perform the requested operation.

User input:

- `Parse` throws `FormatException` with a concise error;
- `TryParse` returns `false` and does not throw;
- an optional detailed API may return a `MusicParseError` with code and position.

There is no global `catch` transforming an error into an empty value, `0`, or silent `null`.

## 12. Compatibility and Versioning

- This API does not seek compatibility with `Enaxos.Harmony.Core`.
- No `Py*` type, general extension method, or historical alias is added.
- The first stable version should be published only after invariants are frozen and the API is reviewed.
- Any change to equality, spelling, or the formula of a standard type is considered breaking.
- Standard definitions have versioned, tested identifiers.

## 13. Historical Implementation Order

This order guided the initial build of the library and remains useful for understanding module dependencies:

1. `NoteLetter`, `Accidental`, `PitchClass`, `SpelledPitch`, `Note`.
2. `Interval`, quality, calculation between notes, and transposition.
3. `FormulaDegree`, mode families, scales, and pentatonics.
4. Key signatures, degrees, and relative/parallel relationships.
5. Chords, realizations, transformation provenance, and harmonic functions.
6. Chord recognition and scale-candidate ranking.
7. Circle of fifths and relationships.
8. UI geometry, French/American terminology, and formatting.
9. Tuning and MIDI adapter.

Each step is covered by tests for the corresponding invariants.
